# Work Management

Shared domain rules. Does not authorize writes by itself. Write only when user/automation scope allows.

## Databases

IDs are hints. Prefer finding by name, then verify schema.
- **Tasks** `collection://1cb5979e-268c-80e9-bd7d-000b00ac4424`: executable actions. Fields: `Name`, `Status`, `Sprint`, `Due`, `Assignee`, `Project`, `Meetings`. `Summary` autogenerated; do not edit.
- **Projects** `collection://4f5bd6fe-452e-4fbc-bcf8-cfcc2d19a2ae`: finishable durable work buckets. Fields: `Name`, `Status`, `Customers`, `Tasks`, `Meetings`, `Target`. `Summary` autogenerated; do not edit.
- **Customers** `collection://2635979e-268c-8191-b322-000bd3109d1c`: stable account info. Fields: `Name`, `Status`, `Domain`, `Projects`, `GitHub Repo URL`, `Moneybird Contact ID`, `Moneybird Contact URL`. `Summary` autogenerated; do not edit.
- **Meetings** `collection://1cb5979e-268c-808d-888d-000bfa3a527c`: source, not work bucket. Fields: `Name`, `Projects`, `Tasks`; `Customers` rollup. `Summary` autogenerated; do not edit.
- **Someday** `collection://8b6245be-419a-4203-97e4-f7660514c661`: brain dump. Ideas, vague initiatives, someday/maybe, no next action. Fields: `Name`, `References`. `Summary` autogenerated; do not edit.

## Status

- Tasks: `Triage`, `Backlog`, `Todo`, `Doing`, `Waiting`, `Done`, `Canceled`.
- Projects: `Discovery`, `Planned`, `In Progress`, `Paused`, `Completed`, `Canceled`.
- Customers: `Prospect`, `Active`, `Lost`, `Churned`.

## Routing

- Task = concrete next action, clear outcome.
- Project = big durable bucket; Linear-style finishable outcome.
- Customer = stable account context.
- Meeting/Slack/email/calendar/WhatsApp = source.
- Someday = non-executable idea/note.
- `Backlog` = executable, not active.
- `Todo` = executable, ready.
- `Doing` = active work, including AI/background work that is ready for Sil to review.
- `Waiting` = blocked on external input, customer/vendor decision, dependency, or timing.
- `Sprint` = active timing; one week, Saturday through Friday.
- `Triage` = inbox signal, not resting status.
- Later but executable -> Task `Backlog`, not Someday.
- Unknown actionability -> keep `Triage`; note missing decision.

## Relations

- Customer/delivery Tasks need `Project`. Standalone ok for personal/admin only.
- Projects usually link `Customer`; internal projects may not.
- Source context should end in Tasks, Projects, Customers, or Someday.
- Durable scope/decisions/customer context -> Project.
- Execution details for one action -> Task.

## Triage Decision

1. Read item + source context.
2. Decide: Task, Project/Customer update, source-only context, or Someday.
3. If Task: concrete `Name`, correct `Status`, `Project` when known.
4. Active soon -> `Todo` + `Sprint` when useful. AI/user actively working or Sil review needed -> `Doing`. Blocked -> `Waiting`. Not active -> `Backlog`.
5. Not executable -> Someday or triage with note.
6. Preserve trace via relations/body.

## Bodies

Read body via Notion markdown/body API before interpreting/changing. Never blind append.
Do not paste full transcripts, duplicate Project scope in Tasks, overwrite execution/review top sections, or edit autogenerated `Summary`.

Task trace pattern. Work-triage/work-management writes belong in dated trace/context, usually below any active execution/review section.

```md
### YYYY-MM-DD - Source/update label

- Captured change/action context.
- Links, dates, people, source page/thread.
- Status/project/sprint reason, if relevant.
```

Project body: loose durable context. Use headings that fit the project; do not force all sections.

```md
## Outcome

Finishable result.

## Scope / agreements

- Commercial, timing, ownership, agreed scope.

## Current context / decisions

- Current state, decisions, blockers, constraints.

## Delivery notes

- Notes that survive across tasks.

## References

Primary source links/mentions.
```

Allowed: add/remove/rename sections when clearer. Keep durable decisions, scope, links, and constraints easy to scan.

## Constraints

- Ask before destructive, irreversible, privacy-sensitive action.
- Use real Notion schema/status names.
- Do not hallucinate fields.
- Compress duplicated old task bodies when migrating to Projects.

## Examples

- "Maybe make a dashboard someday" -> Someday.
- "Renew passport later" -> Task `Backlog`.
