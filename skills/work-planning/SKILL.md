# Work Planning

Use with `work-management`. Use `work-execution` only for Task readiness, evidence, and status-routing rules.

Friday weekly planning is practical. It may look backward only to decide what should happen next. It is not personal reflection and not broad inbox triage.

## Metadata

Before planning, fetch live metadata for Tasks, Projects, Someday, Sprints, Updates, and any related Customers/Meetings needed for context. Use live database/property descriptions, relation targets, and status/select options for schema semantics.

Keep Sprints metadata especially fresh: it defines the Task/Update Sprint relation and the Saturday-Friday planning week.

## Purpose

1. Pick a realistic next Sprint.
2. Promote executable work from Backlog/Someday when it has become real.
3. Add recurring planning items without relying on Notion recurring templates.
4. Create prefilled customer-facing Project Updates drafts for active customer projects when useful.
5. Clean stale open-work statuses when that improves planning.
6. Surface decisions Sil must make.

## Inputs

Fetch and inspect:
- Tasks: open statuses, overdue, due soon, current/next Sprint.
- Projects: active/open statuses.
- Someday: recently edited, recurring themes, or items that now look executable.
- Last week's completed/status-changed Tasks only as practical context for carry-over/follow-up.
- Upcoming Notion Sprint page from the `Sprints` database: `Weekly YYYY-MM-DD`, date range Saturday-Friday. This is the Task/Update `Sprint` relation, not the `Updates` database and not a Reflects page.
- Updates database by name/schema; use this for customer-facing Project Update drafts.
- Meetings/calendar only when needed to understand next-week commitments.

Read body before interpreting or changing a Task, Project, Someday, Sprint, or Updates page. Use real schema/status names. Do not edit autogenerated summary fields.

## Planning Rules

1. Sprint = one week, Saturday through Friday.
2. Move only concrete executable work into next Sprint.
3. Always include open Tasks whose `Due` date falls inside the upcoming Sprint date range: link them to that Sprint and make sure their status is planned/executable unless they are Done, Canceled, or explicitly on hold.
4. Prefer hard due dates, active customer commitments, current Projects, Doing/review items, Waiting items with actionable unblock paths, and work with clear next steps.
5. Move ready Backlog items to ready/todo status and assign next Sprint when they should happen next week.
6. Keep non-urgent executable work in backlog/later status.
7. Keep unclear work in triage status only with a concrete missing decision.
8. Do not manually roll unfinished Notion Tasks from the current Sprint into the next Sprint just because they are unfinished; Notion handles that automatic carry-over.
9. Include practical carry-over from last week only when it needs a real planning decision, status cleanup, or explicit mention.
10. Link planned Tasks and Project Update drafts to the upcoming Saturday-Friday Sprint page.
11. Do not write personal reflection, feelings/energy review, or planner-photo/schrift content. Sil does that on paper.
12. Do not overload the Sprint. If everything is important, the plan is fake.

## Project Updates

Create/update weekly customer update drafts inside the Notion `Updates` database during Friday planning.

Use live Updates metadata for fields/statuses. The page body carries the actual prefilled update message.

Rules:
1. Draft only for active customer-facing Projects with meaningful progress, blockers, decisions, or next steps.
2. Do not create updates for internal projects unless explicitly useful.
3. Prefer one update per Project per Sprint. Dedupe by Project + Sprint before creating.
4. If there is no useful customer-facing signal, skip silently or create skipped status only when preserving history matters.
5. Set new update drafts to live draft status.
6. Set health using judgement, not task counts alone.
7. Set channel from known customer/project preference when available; otherwise use the neutral/manual option.
8. Never send the update externally.

Update body rules:
- No greetings or direct salutation such as "Hi", "Hello", "Hey", or named openers.
- Keep the message channel-neutral so it works for email or Slack.
- Start directly with the status update.
- Use short bullets. Remove filler and throat-clearing.

Good body shape:
```md
- Completed / changed:
- Open / blocked:
- Next:
- Needed from you:
```

Omit empty sections. No fake busyness, no hour-filling language, no promises beyond the evidence.

## Recurring Planning Items

Replace recurring Task templates with planning behavior:
- Weekplanner invullen: personal paper reflection; do not create/manage it as an OpenClaw recurring Task.
- Uren invullen in Productive: include/create a planning Task when a new month has started and the previous month's hours likely need logging.
- Teveo sprint: ensure a Teveo sprint Project exists for the Monday inside the upcoming Saturday-Friday Sprint, linked to the Teveo Customer and relevant Sprint context through related Tasks/Updates where possible. Name it with the sprint number, e.g. `Teveo sprint 20`, inferred from the previous Teveo sprint Project first, then monday/Slack context if needed.
- Fayn sprint: ensure a Fayn sprint Project exists every other week for the Tuesday inside the upcoming Saturday-Friday Sprint, linked to the Fayn Customer and relevant Sprint context through related Tasks/Updates where possible. Name it with the sprint number, e.g. `Fayn sprint 4`, inferred from the previous Fayn sprint Project first, then monday/Slack context if needed. Do not use theme versions like `1.4` as sprint Project names.

Before creating recurring items, dedupe by title, customer, Sprint/week, target/due date, and nearby existing Projects/Tasks. If the Fayn biweekly cadence or naming cannot be inferred, report the decision instead of creating duplicate garbage.

## Someday

During planning:
- Move to Task only when there is a concrete action/outcome.
- Move to Project only when there is a durable finishable bucket.
- Leave as Someday when it is still vague, aspirational, someday/maybe, or taste-dependent.
- Preserve source references when promoting.

## Status Cleanup

Review open Tasks for stale states when planning:
- Active/doing status needs evidence of active work or a concrete `## Review` section for Sil.
- Blocked/waiting status needs a clear dependency/person/date.
- Ready/todo status needs readiness. Missing scope/priority/budget/taste/customer promise means do not pretend it is ready.
- Triage should not rest there. Route clear items; note missing decision for unclear items.
- Closed/done is only for explicitly approved or low-risk fully verified local/admin cleanup.

## Output

Return one concise numbered list:
- Sprint moves.
- Recurring planning items created/included.
- Project Updates drafted/skipped.
- Backlog/Someday promotions.
- Practical carry-over.
- Cleanup fixes.
- Sprint relation/page updates.
- Blockers or decisions Sil must make.

Suppress rediscovered facts. If nothing changed, say that.
