# Sprint Planning

Hermes cron skill. Self-contained for Sprint planning. Use `work-management` only as a global reference when exact shared wording is useful.

Sprint planning is practical, not reflective. Notion automatically closes the previous Sprint and opens the new Sprint; this skill does not need to create or close Sprint pages unless a required page is missing. It fills Sprint bodies, plans Todo/Backlog/Triage Tasks that should be linked into the active/upcoming Sprint, and prepares Notion so `work-execution` can continue with clear Projects, Tasks, and Sprint links. It is not broad inbox triage and not delivery execution.

## Boundary With Triage

1. Do not run `work-triage`, `collect_all_context.py`, incoming lanes, Gmail, Slack, WhatsApp, or broad inbox collectors for normal work-planning runs.
2. Work-planning is Notion-first: inspect Tasks, Projects, Someday, Sprints, and only the narrow source context already linked or summarized there.
3. Use Calendar/meeting context only when needed to understand next-week commitments or an already planned item.
4. If Sil explicitly asks for planning plus inbox triage, run triage as a separate phase and label triage-derived writes separately in the output.
5. If a planning decision seems to require fresh inbox context, report `Decision:` or `Blocked:` instead of silently widening into triage.

## Local Work Model

- Task = concrete executable next action with a clear outcome.
- Project = native real project/commercial delivery bucket: invoiceable project, proposal, prototype, repository, retainer, or durable customer scope.
- Customer = stable account/context layer.
- Someday = non-executable idea, maybe-later note, or vague exploration.
- Sprint = Monday-Sunday planning week.

Tasks may link directly to Customer. Create/link a Project for real project/commercial scope: invoiceable customer work, proposals, prototypes, repositories, retainers, or context that must survive multiple Tasks.

Whenever creating a real Project, create or link at least one concrete Task in the same workflow. If the first Task is unclear, create a scoped planning/scoping Task instead of leaving the Project empty.

Status routing:

- Backlog = executable someday/soon work that is valid to keep, but not ready or important enough for the current Sprint/Todo lane.
- Todo = ready/executable.
- Doing = active work or AI/background work ready for Sil review.
- Waiting = blocked on person, dependency, decision, timing, customer, or vendor.
- Triage = unclear actionability, missing decision, or not routed yet.
- Done/Canceled = only when explicit, or low-risk local/admin cleanup is fully verified.

Read page bodies before interpreting or changing them. Never blind append. Do not edit autogenerated summary fields.

## Execution Contract

1. Finish the full planning workflow in one run.
2. Do not return partial collection as success.
3. If a required planning area, schema, or write cannot be completed, finish with a `Blocked:` or `Failed:` output item and explain the practical consequence.
4. Read page bodies before interpreting or changing Tasks, Projects, Someday, or Sprints.
5. Prefer updating existing Notion items over creating duplicates.
6. Use live schema/status/property names. Do not edit autogenerated summary fields.
7. Do not send external customer messages.
8. Do not do destructive, money/account-impacting, or customer-impacting actions without explicit approval.
9. If a Notion ID is already a data source ID, query it directly instead of failing on `datasources resolve`. Do not surface recoverable schema probing noise unless it affected the result.
10. Do not execute delivery work. Create, link, scope, route, and prepare Notion items so loop-driven work execution can pick them up.

## Sprint Body Workflow

1. Sprint weeks run Monday through Sunday.
2. For cron runs, derive the previous and active/upcoming Sprint from the cron run start time, not from hardcoded assumptions.
3. Fetch the previous Sprint and active/upcoming Sprint pages from the Sprints database. Notion normally creates/closes these automatically; only create a Sprint page if the expected page is missing and the schema clearly supports safe creation.
4. Fill the previous Sprint body with a compact practical review based on linked Tasks, completed/status-changed Tasks, carry-over, blockers, and notable work movement. This is work review only, not personal reflection.
5. Fill the active/upcoming Sprint body with a compact plan: focus, selected Tasks, blockers/decisions, and known constraints.
6. Link planned Tasks to the active/upcoming Sprint relation.

Preferred previous Sprint body section:

```md
## Review

- Completed: concise list or summary.
- Carried over: tasks that still matter and why.
- Blocked/waiting: blockers or decisions needed.
- Notable changes: customer/project/work movement worth remembering.
```

Preferred active/upcoming Sprint body section:

```md
## Plan

- Focus: short practical theme.
- Planned: selected Tasks/work packages.
- Watch: blockers, decisions, deadlines, capacity risks.
```

## Inputs

Fetch and inspect from Notion:

1. Open Tasks: Triage, Backlog, Todo, Doing, Waiting, overdue, due soon, current Sprint, next Sprint.
2. Todo, Backlog, and Triage Tasks that may need Sprint planning now.
3. Active Projects: Discovery, Planned, In Progress, Paused, or edited recently.
4. Someday items that were created/edited recently, recur as themes, or now look executable.
5. Last week's completed/status-changed Tasks only for practical Sprint review, carry-over, or follow-up.
6. Previous Sprint page and active/upcoming Sprint page, including existing linked Tasks.
7. Calendar/meeting context only when needed to understand next-week commitments.

Do not fetch broad Gmail, Slack, WhatsApp, or inbox context here. Those belong to `work-triage`.

## Planning Rules

1. Move only concrete executable work into the upcoming Sprint.
2. Always include open Tasks whose Due date falls inside the upcoming Sprint date range, unless Done, Canceled, or explicitly on hold.
3. Prefer hard due dates, active customer commitments, current Projects, Doing/review items, actionable Waiting items, and work with clear next steps.
4. Move ready Backlog items to Todo/Sprint only when they should happen next week.
5. Leave non-urgent executable work in Backlog.
6. Leave unclear work in Triage only with the exact missing decision.
7. Do not manually roll unfinished current-Sprint Tasks into next Sprint just because they are unfinished; Notion handles carry-over.
8. Include carry-over only when it needs a real planning decision, cleanup, or mention.
9. Do not overload the Sprint. If everything is important, the plan is fake.

## Recurring Planning Items

Replace recurring Notion templates with planning behavior:

1. Weekplanner invullen: personal paper reflection; never create/manage it.
2. Productive hours: should be visible as a Notion Task such as `Productive hours reconcile`, not hidden in a Hermes cron. Sprint planning may link an existing Productive hours Task into the Sprint when relevant. Create a new Productive hours Task only when the weekly visible Task is missing and it is clearly needed; avoid duplicating legacy variants like `Uren invullen in Productive`.
3. Teveo sprint: represent the weekly Teveo client sprint as a Task named like `Teveo sprint 24` / `Teveo sprint 25`. Link the Task to the Teveo Customer and Sil's upcoming Sprint. Infer the next sprint number from previous Teveo sprint Tasks first. Only use monday/Slack context if it is already captured in Notion or Sil explicitly asks for triage-assisted planning. Put selected tickets/current execution state in the Task body.
4. Fayn sprint: do not automatically create biweekly Fayn sprint Projects or Tasks for now. Only update/link existing Fayn sprint work, or create it when explicitly requested by Sil. Do not use theme versions like `1.4` as sprint names.
5. Dedupe recurring items by title, customer, sprint/week, target/due date, and nearby Projects/Tasks.
6. If cadence or naming cannot be inferred, report a decision item instead of creating duplicate garbage.

## Someday

1. Promote to Task only when there is a concrete action/outcome.
2. Promote to Project only when there is a durable finishable bucket.
3. Leave as Someday when still vague, aspirational, someday/maybe, or taste-dependent.
4. Preserve source references when promoting.

## Status Cleanup

1. Doing needs evidence of active work or a concrete Review section for Sil.
2. Waiting needs a clear dependency/person/date.
3. Todo needs scope/readiness; missing scope, priority, budget, taste, or customer promise means not ready.
4. Triage should not rest there. Route clear items; note missing decisions for unclear items.
5. Done is only for explicitly approved or low-risk fully verified local/admin cleanup.

## Output Contract

Return exactly one concise numbered list.

One numbered item = one concrete action, change, decision, failure, or blocker.

Use these compact prefixes:

- `Sprint:`
- `Task:`
- `Project:`
- `Recurring:`
- `Someday:`
- `Cleanup:`
- `Decision:`
- `Blocked:`
- `Failed:`

Preferred format:

```md
1. Type: [Name](link) concrete change/result.
2. Type: [Name](link) concrete change/result.
```

Rules:

1. Include only actual writes, created pages, linked Sprint items, status changes, cleanup fixes, explicit skips that prevented unsafe writes, blockers, or decisions for Sil.
2. Suppress rediscovered facts and internal mechanics.
3. Do not mention tool paths, data source IDs, cron internals, or diagnostics unless they changed the outcome.
4. If triage was explicitly requested as a separate phase, label triage-derived writes separately so planning and triage do not blur together.
5. If nothing changed, output exactly one numbered item: `1. Sprint planning: no planning changes needed.`
